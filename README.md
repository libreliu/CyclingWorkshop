# CyclingWorkshop

CyclingWorkshop 是一个本地 Web 工具，用于把 FIT 运动数据叠加到骑行、跑步或运动相机视频上，生成带速度、心率、功率、路线地图等信息的成片。

效果参考：[159km，从深圳大铲湾骑到惠州双月湾，看海](https://www.bilibili.com/video/BV1FwdUBkEhB/)

## 主要功能

- 读取 FIT 文件并解析轨迹、速度、心率、踏频、功率、海拔、坡度等数据。
- 加载一个或多个视频，并为每个视频单独设置时间同步。
- 在浏览器中编辑叠加组件位置和样式，支持仪表盘、计时器、距离、海拔图、路线地图等组件。
- 支持单视频或批量视频渲染。
- 支持地图瓦片代理、缓存、预下载和缓存覆盖范围可视化。
- 支持软件编码和部分硬件编码器，支持仅导出透明 Overlay 层。

## 环境准备

1. 克隆仓库并进入项目目录：

```bash
git clone <repo-url>
cd CyclingWorkshop
```

2. 创建虚拟环境并安装 Python 依赖：

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
```

macOS / Linux 可使用：

```bash
python -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements.txt
```

3. 准备 FFmpeg / FFprobe。

推荐把 `ffmpeg` 和 `ffprobe` 加入系统 `PATH`。如果没有加入 `PATH`，可以通过环境变量指定路径：

```powershell
$env:FFMPEG_PATH="<path-to-ffmpeg.exe>"
$env:FFPROBE_PATH="<path-to-ffprobe.exe>"
```

macOS / Linux：

```bash
export FFMPEG_PATH="<path-to-ffmpeg>"
export FFPROBE_PATH="<path-to-ffprobe>"
```

4. 启动本地服务：

```powershell
python app.py
```

默认访问地址是 `http://127.0.0.1:5000`。

## 基本使用流程

1. 加载 FIT 文件。

在步骤 1 中选择运动手表导出的 `.fit` 文件。加载后可以检查轨迹、统计信息和异常点。

2. 加载视频。

在步骤 2 中添加一个或多个视频。批量项目中，每个视频可以拥有独立的时间同步和输出路径。

3. 设置时间同步。

优先使用自动推断；如果视频起点和 FIT 起点不一致，可以手动调整偏移，或使用关键帧方式对齐视频时间和 FIT 时间。

4. 设计叠加层。

在步骤 4 中选择模板或手动调整组件。使用 `MapTrack` 地图组件时，建议先完成瓦片缓存，避免预览或渲染时临时下载瓦片。

5. 渲染导出。

在步骤 5 中设置编码器、CRF、音频、输出路径和批量并发数，然后开始渲染。渲染日志可以按任务查看。

## 瓦片缓存与地图注意事项

如果使用路线地图底图，请在正式渲染前预先缓存瓦片。这一点很重要：渲染过程中如果临时请求在线瓦片，网络慢、代理不可用或瓦片服务限流都可能导致地图空白、预览变慢或渲染流水线等待超时。

推荐做法：

- 先加载 FIT 文件，让系统知道轨迹范围。
- 打开顶部的“瓦片管理”。
- 如处在公司网络、校园网或受限网络中，先配置并测试代理。
- 使用“瓦片预下载”为当前 FIT 轨迹下载所需瓦片。
- 或在“缓存可视化”地图中按住 `Shift` 拖拽选择区域，再下载选区。
- 确认缓存统计和覆盖范围正常后，再开始最终渲染。

默认瓦片缓存目录是 `CyclingWorkshop/tile_cache`。如果地图组件仍显示空白，优先检查瓦片样式是否匹配、代理是否可用、选区是否覆盖完整轨迹。

## 批量渲染建议

批量并发数默认建议保持为 `1`。每个视频任务内部还会启动 Decode、Overlay、Encode 等子进程，并分配共享内存；并发数调到 `2` 或更高时，实际资源占用会成倍增长。

使用批量渲染时请注意：

- 每个视频必须使用不同的输出路径，避免一个任务完成后的音频混流、重命名或写文件影响另一个任务。
- 首次排查问题时，把“批量并发数”设为 `1`，确认串行渲染稳定后再尝试提高。
- 如果保留原始音频，渲染完成后还会执行音频混流，磁盘 I/O 会明显增加。
- 如果遇到完成一个视频后其它视频变慢、卡住或报错，可以先改为“无音频”测试。
- 硬件编码器通常不适合高并发；GPU 编码资源有限时，优先降低并发或改用 `libx264`。
- 4K 视频会显著增加共享内存、Pillow 图像处理和编码压力，建议先用短片段验证配置。

## 渲染参数建议

- 编码器：兼容性优先选 `libx264`。
- CRF：数值越低质量越高、文件越大；常用范围是 `18-23`。
- 预设：`fast` 或 `medium` 是比较稳妥的起点。
- 音频：如不需要保留原声，选择“无音频”可以减少收尾阶段的混流时间。
- 渲染模式：默认使用“多进程流水线”。“Tick 单线程调试”主要用于排查问题，速度较慢。
- Overlay-only：只导出透明叠加层，适合后期软件中再合成。

## 常见问题

### 地图组件渲染慢或空白

通常是瓦片没有缓存、代理不可用、瓦片服务请求失败或选错了地图样式。先到“瓦片管理”中预下载轨迹区域，并确认缓存覆盖范围。

### 批量渲染互相影响

优先检查输出路径是否重复，然后把批量并发数降为 `1`。如果问题只在保留音频时出现，可以改为“无音频”验证是否由音频混流阶段引起。

### 渲染超时或子进程异常

当前流水线中，共享内存槽等待通常是 `30` 秒，子进程收尾等待是 `10` 秒。资源被其它并发任务占满、瓦片临时下载、磁盘 I/O 拥堵都可能触发类似问题。

### FIT 数据和视频不同步

先确认视频是否经过剪辑、变速或抽帧。如果只是起点不同，用 offset 调整即可；如果中途速度不一致，使用关键帧同步会更可靠。

## 项目目录

- `app.py`：Flask 应用入口。
- `api/`：FIT、视频、渲染、项目、瓦片等接口。
- `services/`：FIT 解析、视频分析、帧渲染、渲染流水线、瓦片缓存等核心逻辑。
- `static/`：前端 JS/CSS 和第三方静态资源。
- `templates/`：Web UI 页面。
- `saved_states/`：本地项目保存目录。
- `tile_cache/`：地图瓦片缓存目录。
- `output/`：默认输出目录。
